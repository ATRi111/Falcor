VoxelizationGPU tool brief

Purpose
- New executable: build\windows-vs2022\bin\Release\VoxelizationGPU.exe
- It does not reimplement GPU voxelization. It directly drives the existing VoxelizationPass_GPU and still uses the original VoxelizationPass::write() path, so output format/naming stays on the original code path.

Build
- General build instructions are in:
  .FORAGENT\build_any_target_brief.txt
- This tool target name is:
  VoxelizationGPU
- Typical binary path after a Release build:
  build\windows-vs2022\bin\Release\VoxelizationGPU.exe

Usage
- Basic:
  build\windows-vs2022\bin\Release\VoxelizationGPU.exe <scene.pyscene>
- Example:
  build\windows-vs2022\bin\Release\VoxelizationGPU.exe Scene\Arcade\Arcade.pyscene --voxel-resolution 8 --sample-frequency 64 --solid-rate 1.0 --max-frames 500 --timeout-seconds 120
- Common options:
  --device-type d3d12|vulkan
  --gpu <index>
  --voxel-resolution <n>
  --sample-frequency <n>
  --polygon-per-frame <n>
  --scene-name <Auto|custom>
  --lerp-normal
  --solid-rate <float>
  --max-frames <n>
  --timeout-seconds <n>
  --list-gpus

Parameter meaning
- voxel-resolution: Controls voxel size and final grid dimensions. This changes the output.
- sample-frequency: Rays per direction used during polygon/area estimation. Higher is slower but usually more stable. This changes the output.
- lerp-normal: Uses interpolated vertex normals instead of sampling the normal map. This changes the output.
- polygon-per-frame: Max clipped polygons processed per batch/frame. Mainly affects runtime scheduling and speed, not intended final output.
- solid-rate: GPU-only estimated solid-voxel ratio used for buffer sizing. Mainly affects VRAM use and overflow risk, not intended final output if large enough.
- scene-name: Mainly affects output filename prefix and ImageLoader scene mapping. It does not choose which scene file gets loaded.

Behavior
- Headless run.
- Loads the scene, creates a minimal render graph with VoxelizationPass_GPU, auto-starts generation, writes the .bin file, prints the output path, then exits.
- Output is still written under resource\ by the original pass code.

Important note
- Default GPU pass behavior is unchanged. The original VoxelizationPass_GPU buffer sizing can still fail for some scenes/resolutions when solidRate is too small.
- Default solidRate has been raised from 0.05 to 0.4 as a lightweight mitigation.
- This is still not a hard safety fix. Some scenes/resolutions may still need a larger explicit --solid-rate.
