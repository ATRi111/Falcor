1. 结论摘要

问题 最新结论 说明
是否进行体素化 支持 有两条入口：VoxelizationPass_CPU 和 VoxelizationPass_GPU。
是否进行体素渲染 支持 最终由 RayMarchingPass 的全屏像素着色器做体素光线步进与着色。
当前是否直接渲染 mesh 在体素方案内不直接渲染 体素脚本里没有 VBufferRT / GBufferRT / PathTracer；mesh 只作为体素化输入数据。
仓库里是否存在独立 mesh 渲染 存在 例如单独的 scripts/PathTracer.py，但它与体素图是分离的。
当前是否做 mesh + voxel 混合渲染 没有 没有看到把 mesh 图像与体素图像混合、合成或分层显示的现行通路。
是否完全脱离原始场景 mesh 没有 原始 mesh 仍用于体素化、材质采样、场景灯光和自发光三角形采样，但这属于数据来源复用，不是 mesh 直接成像。
运行模式 缓存驱动 先生成 .bin 体素缓存，再由 ReadVoxelPass 读入，之后用 RayMarchingPass 渲染。
当前最准确的表述是：这是一个以 mesh 为输入、以 voxel cache 为中间表示、以 ray marching 为最终成像手段的体素渲染方案， 而不是一个正在把 mesh 图像和 voxel 图像一起混合输出的方案。

2. 当前渲染链路

当前 scripts/Voxelization_CPU.py 和 scripts/Voxelization_GPU.py 的图都只有同一条主链：

voxel_pass = createPass("VoxelizationPass_CPU")   # 或 VoxelizationPass_GPU
read_pass = createPass("ReadVoxelPass")
marching_pass = createPass("RayMarchingPass")
viewport_pass = createPass("RenderToViewportPass")
accumulate_pass = createPass("AccumulatePass")
tone_mapper = createPass("ToneMapper")
flowchart LR
SceneMesh[Scene Mesh / Material / Light] --> Voxelize[VoxelizationPass_CPU or GPU]
Voxelize --> Cache[resource/*.bin voxel cache]
Cache --> Read[ReadVoxelPass]
Read --> March[RayMarchingPass]
March --> View[RenderToViewportPass]
View --> Acc[AccumulatePass]
Acc --> Tone[ToneMapper]
阶段 当前职责
VoxelizationPass_* 从场景 mesh 生成体素缓存，输出到仓库相对路径 resource/。
ReadVoxelPass 读取 .bin 文件，把缓存拆成 vBuffer、PrimitiveBSDF、Ellipsoid、blockMap。
RayMarchingPass 按像素发射相机光线，在体素网格里做 DDA 或 block-map 跳步，并计算直接光和多次反弹。
RenderToViewportPass 只负责把输入纹理拷到输出视口，不做与其他 mesh 结果的混合。

补充：当前头部提交 08e5552 的作用主要是把体素缓存路径改成相对仓库根目录解析，不改变核心算法。

3. 当前体素化方案

3.1 共享基础

VoxelizationBase::UpdateVoxelGrid() 先用场景 AABB 求出统一体素网格。分辨率由 UI 里的 Voxel Resolution 控制，网格边长按场景最长轴做归一化；缓存文件写入到仓库相对路径 resource/。

共享数据 作用
vBuffer 3D 体素到稀疏实体体素索引的映射。
VoxelData 每个实体体素的聚合材质 ABSDF、拟合椭球、投影面积球谐函数。
blockMap 稀疏占据信息，用于 ray marching 的块级跳步。
PolygonBufferGroup 按 maxPolygonCount 分组管理变长 polygon buffer，对应 UI 中的 Polygon Per Frame。

3.2 CPU 体素化路径

  1. LoadMesh.cs.slang 从场景读取顶点、法线、UV、索引。
  2. 这些数据被回读到 CPU。
  3. MeshSampler::clipAll() 在 CPU 上把每个三角形裁到体素单元，形成每体素的 polygon 列表。
  4. 公共的 AnalyzePolygon.cs.slang 再在 GPU 上对每个实体体素做材质采样、椭球拟合、投影面积估计和 blockMap 更新。
  5. 结果写成 .bin 缓存文件。

3.3 GPU 体素化路径

  1. SampleMesh.cs.slang 先在 GPU 上遍历 mesh 三角形，做 box-clip，并对命中的体素原子分配偏移。
  2. 这一阶段直接累计体素的 ABSDF 面积、材质统计，以及每个体素的 polygon 数量。
  3. 然后把 solidVoxelCount、vBuffer、polygonCountBuffer 回读到 CPU，仅用于给变长 polygon buffer 做分组预分配。
  4. ClipMesh.cs.slang 再在 GPU 上把真正的 polygon 写入每组 buffer。
  5. 之后仍走公共的 AnalyzePolygon.cs.slang，补齐椭球、投影面积球谐函数和 blockMap。
  6. 最后同样写成 .bin。
所以当前 GPU 路线并不是“全流程零回读”的纯 GPU 管线。它已经把大头工作搬到 GPU，但中间仍有一次 CPU 参与，用来处理 PolygonBufferGroup.reserve() 的分组和容量规划。

3.4 最新状态下的关键特征

特征 最新状态
法线/材质来源 来自原始 mesh 的顶点法线、UV 和材质纹理。
法线聚合 通过 ABSDF 把体素内多个多边形压成少量 lobe。
几何近似 每个实体体素额外拟合一个 Ellipsoid，供命中判定和调试使用。
稀疏性利用 使用 vBuffer 稀疏索引和 blockMap 跳块。
生成节奏 按 Polygon Per Frame 分批执行，不是一次性把所有 polygon 全吞完。

4. 当前体素渲染方案

4.1 运行时输入

ReadVoxelPass 从缓存文件中读入 GridData、vBuffer、VoxelData、 blockMap，再通过 PrepareShadingData.cs.slang 把 VoxelData 拆成运行时使用的 PrimitiveBSDF 和 Ellipsoid。

4.2 光线步进

RayMarching.ps.slang 对每个像素：

  1. 从当前相机逆 VP 矩阵反投影出相机射线。
  2. 先与体素网格包围盒求交。
  3. 在网格内部用 DDA 或 blockMap mip-style 跳步搜索命中体素。
  4. 如果启用 CHECK_ELLIPSOID，会在命中体素后再用椭球裁段做更细命中。
  5. 拿到命中体素的 PrimitiveBSDF 后，计算直接光与多次反弹。

4.3 着色模型

模块 当前做法
PrimitiveBSDF 由体素内聚合后的表面 BRDF、覆盖率、内部可见性和投影面积函数组成。
直接光 由 sampleLight() 混合采样一般灯光、环境光、自发光三角形。
阴影/遮挡 不是查 mesh，也不是 RT；而是继续在体素网格中做 rayMarching()。
全局光照 在体素命中点之间做非递归多次反弹，默认 UI 是 Max Bounce = 3，着色器上限为 4。
背景 可选环境贴图或纯色背景。
调试模式 支持显示法线、椭球、投影面积、覆盖率、NDF 选中体素等。
这里最重要的一点是：当前的可见性判断和反弹路径都发生在体素表示里。即使灯光来源能复用原始场景的灯和自发光三角形， 阴影与遮挡依然是 voxel-space 的。

5. Mesh 渲染与混合渲染判断

5.1 有没有 mesh 渲染

要分成两个层次看：

层次 判断 理由
ATRi111 当前体素方案内部 没有 mesh 直接成像 scripts/Voxelization_CPU.py / scripts/Voxelization_GPU.py 只创建体素相关 pass，没有创建 VBufferRT、GBufferRT、PathTracer。
整个仓库级别 有独立 mesh 渲染 scripts/PathTracer.py 等脚本仍然是标准 mesh 路径，但它们和体素图分开使用。

5.2 有没有混合渲染

当前没有看到 mesh + voxel 的混合渲染方案。

检查点 结果
体素脚本是否引入 mesh 渲染 pass 没有。
体素像素着色器是否发真实 RT 射线到 mesh 没有检到 traceRay / TraceRay。
RenderToViewportPass 是否做结果合成 没有，只是把输入纹理采样到输出。
是否有“前景体素、背景 mesh”之类双通路 没有现行脚本或 pass 连接证明这一点。

5.3 但它确实复用了哪些 mesh 信息

  • mesh 几何：用于体素化时读取三角形、法线、UV、材质。
  • mesh 材质：用于体素内的纹理采样和 ABSDF 聚合。
  • 场景灯光：Point、Directional、EnvMap 直接参与 voxel shading。
  • 自发光三角形：用于作为光源被采样，命中时还会做近邻 emissive 检查。
因此如果一定要说“混合”，那也只是mesh 数据参与 voxel 着色，而不是mesh 图像与 voxel 图像混合显示。 从成像路径上看，当前方案仍然是纯 voxel ray marching。
